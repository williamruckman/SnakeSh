Security and Secrets

SnakeSh is designed to keep connection metadata usable without treating credentials casually. Session data is encrypted at rest and can work with a dedicated secrets backend when you need stronger separation between connection metadata and password storage.

Encrypted Session Store

Session data is encrypted locally with Fernet-based symmetric encryption. The encryption key is generated once and stored in the configured secrets backend or the default platform credential store when appropriate.

Secrets Backends

Supported backends include OS keyring, 1Password CLI, Bitwarden CLI, Keeper Commander, KeePassXC CLI, and HashiCorp Vault KV v2. Backend-specific setup and validation are available from Settings.

Startup Gate

SnakeSh can require a master password before opening the main interface. That startup gate is configured in Settings and stores only the derived verification material needed for the prompt, not a plaintext password.

Host and Certificate Trust

  • SSH and SFTP host keys must be explicitly trusted before reuse.
  • Linux RDP launches can prompt for certificate trust before the viewer starts.
  • Telnet over TLS can apply certificate-validation controls when configured.

Exports and Logs

Exports can be password-protected so backup or transfer bundles are not left in plain text. Terminal logs and Syslog / SNMP monitor captures are useful operationally, but they may contain sensitive command output, network identifiers, trap payloads, or device metadata, so review your retention settings and storage location carefully. Exported settings and sessions do not embed passwords or backend auth tokens that remain stored in the OS keyring or external secrets backend, and monitor history archives are not bundled into first-party exports.

Operational Recommendations

  • Prefer SSH keys over passwords when the target systems allow it.
  • Use a secrets backend for reusable passwords and API tokens.
  • Review auto-logging behavior if commands may print secrets.
  • Treat captured syslog messages and SNMP trap history as sensitive data when they contain operational or customer identifiers.
  • Use encrypted exports when sharing or archiving data outside the current machine.

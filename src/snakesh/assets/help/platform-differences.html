Platform Differences

SnakeSh keeps the session library, workspace model, and most terminal workflows consistent across operating systems, but some capabilities change because the host OS exposes different viewers, shells, packaging models, and privilege rules.

What Stays the Same Everywhere

  • SSH, SFTP, Telnet, Serial, and Local Shell tabs use the same terminal workspace model.
  • Local Shell tabs can detach and reattach the same way as other terminal tabs.
  • Fast Commands on RDP, VNC, and NoMachine tabs copy to the clipboard instead of injecting keystrokes.
  • Network Inspector, File Hash, OUI Lookup, Web Server, Syslog / SNMP Monitor, and the rest of the Tools menu follow the same basic GUI pattern.
  • Session encryption, saved-session structure, and secrets-backend support are shared across platforms.

Windows

Local and Terminal Work

Local Shell tabs normally start with PowerShell or cmd. SSH, SFTP, Telnet, Serial, logging, grouping, and profiles behave the same way they do on other platforms.

Remote Desktop Workflow

RDP launches through mstsc. This is a Windows-native external viewer workflow rather than the Linux in-tab xfreerdp path. VNC depends on an installed compatible viewer. NoMachine uses nxplayer in detached mode.

X11 and Helpers

Windows is the platform where SnakeSh provides the most direct X11 helper behavior. If you enable SSH X11 forwarding, SnakeSh can prefer VcXsrv, detect other X servers, and offer install or launch assistance when needed.

Web Server Tool

Windows does not need the same special low-port elevation branch that Unix-like systems do for the Web Server tool. The GUI still manages the server instance, logs, and start or stop behavior in the same way.

Syslog / SNMP Monitor

Windows does not use the same low-port elevation flow that Linux and macOS do for monitor listeners. The same GUI still manages profile settings, local storage, dashboard graphs, and archive retention.

Linux

Packaging and Integration

Linux builds are distributed as an AppImage, and SnakeSh can create or repair desktop integration from Settings. This is the most desktop-integration-aware platform in the current app.

Remote Desktop Workflow

Linux has the richest RDP behavior in SnakeSh because xfreerdp can be used for a supported in-tab workflow or a detached viewer workflow. Linux RDP also includes a GUI certificate-trust prompt before launch when needed.

Viewer and Network Tooling

Linux commonly has the broadest set of optional helper tools available, including xfreerdp, ip, arp, resolver tools, and multiple VNC viewer options. As a result, the Network Inspector and protocol launchers often expose the most complete output here.

X11 and Privileged Ports

SSH X11 forwarding fits naturally into a Linux desktop that already has a working X environment. For the Web Server tool and the Syslog / SNMP Monitor, privileged ports can trigger GUI elevation through pkexec.

macOS

Packaging and First Launch

Unsigned local macOS builds are supported, and signed/notarized release artifacts depend on configured release credentials. Unsigned builds can involve Gatekeeper prompts or a Finder Open workflow before the app is allowed to run.

Remote Desktop Workflow

RDP is not currently supported on macOS in SnakeSh. SSH, SFTP, VNC, NoMachine, Telnet, Serial, and Local Shell workflows remain available. VNC and NoMachine rely on installed external viewer tools where appropriate.

Terminal and Tool Behavior

Local Shell tabs use the detected macOS shell environment. The Tools menu remains available, but output from network and resolver inspection can differ from Linux because macOS uses different underlying system commands and data sources.

Web Server Tool

When the Web Server tool needs a privileged port on macOS, SnakeSh uses an administrator-prompt style elevation path rather than a terminal-based sudo flow.

Syslog / SNMP Monitor

When the Syslog / SNMP Monitor needs a privileged port on macOS, SnakeSh uses the same administrator-prompt style GUI elevation path instead of asking you to open a shell.

How to Think About Cross-Platform Differences

  • If the feature is purely inside the SnakeSh terminal workspace, behavior is usually the same everywhere.
  • If the feature launches an external viewer, behavior depends heavily on the host OS and installed client tools.
  • If the feature inspects local networking, expect the same categories of data but not necessarily identical fields or formatting.
  • If the feature needs system integration or elevation, Linux and macOS usually need a GUI privilege path while Windows follows native Windows application behavior.

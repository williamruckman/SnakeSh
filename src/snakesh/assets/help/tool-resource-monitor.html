Tool: Resource Monitor

The Resource Monitor is SnakeSh's live local-system telemetry view. It opens in a separate non-modal window, keeps a rolling in-memory history from the moment you launch it, and clears that history when you close the tool.

Overview Tab

The Overview tab is designed as a Mission Center-style dashboard for quick visual scanning.

  • Summary cards show current CPU, memory, disk, per-device disk I/O, network bandwidth, process count, and per-adapter GPU status.
  • Charts track the last live samples collected while the window remains open.
  • Filesystems shows accessible local partitions, highlights the filesystem that contains your home directory, and includes per-device read or write throughput when SnakeSh can resolve it.
  • Interfaces shows detected network interfaces with separate IPv4 and IPv6 addresses, state, link speed, and current receive or send rates, and can include offline adapters when that Settings option is enabled.

GPU telemetry is best effort. SnakeSh attempts to detect NVIDIA, AMD, and Intel adapters and shows whichever utilization, VRAM, and temperature metrics are available for each adapter on the current OS and driver stack. When only partial GPU telemetry is available, the GPU cards stay readable and show which metrics are unavailable instead of treating the whole tab as failed.

Disks and GPU Tabs

The Disks tab keeps the home-volume space summary, but the live throughput cards, mounted-filesystem table, and throughput chart are shown per resolved device rather than as one machine-wide aggregate. The GPU tab keeps the existing card layout while rendering one line per adapter for utilization, VRAM, temperature, and adapter status above the per-adapter history charts.

Network Tab

The Network tab shows detected wired, wireless, virtual, and loopback adapters. When Offline Adapters is enabled in Settings, offline adapters stay visible with an Offline state until they reconnect or are removed from the system.

Settings Tab

The Settings tab controls Resource Monitor display and refresh behavior for future samples and persists the choices in SnakeSh settings.

  • Offline Adapters: toggles whether offline network adapters stay visible.
  • Zoom: scales Resource Monitor cards, tables, and charts between 75% and 150%.
  • Telemetry Refresh: controls the live overview sample interval from 500 ms to 10000 ms.
  • Process Refresh: controls process-list refresh timing from 2000 ms to 30000 ms.
  • Chart History: keeps between 2 and 60 minutes of rolling in-memory chart samples while the tool is open.

Processes Tab

The Processes tab focuses on the live process list for the current machine.

  • Sort by CPU, memory, thread count, start time, or any of the visible process fields.
  • Use the search box to filter by process name, PID, user, or command text.
  • Use End Task for a normal termination request or Force Kill for a stronger stop action.
  • SnakeSh blocks attempts to terminate its own GUI process from this tool.

Authorization Prompts

If a selected process requires elevated privileges to stop, SnakeSh first tries the normal process action and then offers the same OS-native authorization flow used by other privileged tools. On Linux this can use pkexec or sudo, on Windows it can trigger a UAC prompt, and on macOS it can use an administrator-authorization dialog.

What Does Not Persist

The Resource Monitor does not save chart history, process filters, or other live session state after the tool closes. Display and refresh preferences from the Settings tab are saved, but each new launch starts a fresh in-memory monitoring session.

Freeze Diagnostics

If you are investigating a Windows freeze or long-running hang around the Resource Monitor, launch SnakeSh with --debug-level debug before reproducing the issue. That per-run diagnostics mode writes a session log under %LOCALAPPDATA%\SnakeSh\logs\debug\YYYY\MM\ unless you override it with --debug-log-file. The log captures worker timing, warnings, and hang watchdog traceback dumps, while the Settings crash-log toggle remains focused on fatal crashes and native faults.

  • Start snakesh --debug-level debug.
  • Open the Resource Monitor and reproduce the hang.
  • After the freeze or forced close, collect the newest diagnostics log from the debug log folder.
  • If the UI stops pumping events for roughly 15 seconds, the same log should include an all-thread traceback dump.
